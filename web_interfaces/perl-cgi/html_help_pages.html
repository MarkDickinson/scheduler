Job Management

How to make a job run NOW, Asking a job to run immediately

From the job status overview screen you can select the [Actions] button to obtain a list of possible commands you can issue for a job on the active queue.

The [runnow] button from this screen will alter the scheduled runtime of the current job to the current time, making it immediately available to be scheduled on, assuming the following conditions are met.

  • Job is not in a failed state
  • job is not in a dependency state waiting for other jobs to complete
  • the job is not held

Note that even if all conditions are met the job may not immediately start running, it will be scheduled onto the run queue as available to run imediately, but may wait depending on current scheduler availablility which may be such things as a NULL* (exclusive) job running or the maximum jobs already running. It will be run when the scheduler is able to start the job.

Holding a job

From the job status overview screen you can select the [Actions] button to obtain a list of possible commands you can issue for a job on the active queue.

From this screen you can click on the button to hold the job. The effect will be that the job goes into a hold state and will not automatically run at the time due. It will not run until the hold state has been cleared (see below).

If you hold a 'repeat'ing job, it will not attempt to catch up any missing intervals.

Releasing a job

From the job status overview screen you can select the [Actions] button to obtain a list of possible commands you can issue for a job on the active queue.

One off the options is to clear the hold flag from the selected job. The effect of this is to requeue the job onto the active queue at the time it was origionally scheduled to run.

It is important to note that 'repeating' jobs do not 'catchup'. If a job to run every 15 minutes has been held, then released; it will run at the original time and then reschdule to the next time after the current time, it will not attempt to catch up.

Deleting a job

Deleting a job is a systems administrator function.

Deleteing a job should only be done if this job is to also be deleted from the job definition database immediately after the job is deleeted from the active queue. Or more bluntly only ever delete a job if it is never expected to run again. Failure to adhere to this rule can result in time scheduling becoming corrupted.

As such it is not available on the browser interface, it is something you can talk to you system administrator about.

Alert Management

Acknowledging an Alert

From the 'Active Tasks' menu option you can chose to display the current alert queue. One of the buttom options presented there is to acknowledge the alert.

Note that this only changes the colour of the message when users view the alerts screen, it remains red on the job status screen, simply beacause an alert is an alert, and is treated as such on that screen.

Restarting a failed job

From the 'Active Tasks' menu option you can chose to display the current alert queue. One of the buttom options presented there is the option to restart a job. It is expected that the problem that caused the job to fail has been corrected before the job is restarted.

Note that the job will start running from the begining of the script. The scheduler is designed to work on discrete work units and dependencies, if some of your jobs do massive lots of work that havn't been split out; tough. The job will restart from the beginning.

Forcing a job from failed to completed state (into OK status)

This option has not been provided in the browser interface. It is available from the command line interface for systems administration use.

Your system administrators can review the job log to determine if a job needs rerunning, or is OK. If they decide it's OK they can force the alert to be cleared.
This option is not available to point and click as the job logs need to be reviewed before deciding this is an option, your system administrators can force the job to an OK state, so talk to them.
